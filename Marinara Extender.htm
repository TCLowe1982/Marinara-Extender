Here's what's currently in the wireframes vs. what's missing:

Currently implied, but not the star:

  • Sketch 1's YAML snippets show index.yaml files in each scope — but they look like content files, not lookup tables.
  • Sketch 3's loader walks layers, but it's drawn as if it reads full layers wholesale.

What the roadmap actually does (and what's missing):

  • A flat, fast-to-scan index per scope (e.g. global/index.yaml) that lists: every topic/entity/file that exists in that scope, where to find it, a one-line summary, and a token-cost estimate.
  • Loader reads indexes first (cheap — a few hundred tokens total). Then it pulls only the specific entries it needs by ID.
  • "What's the user's wife's name?" → look in index → see entity: wife → character/entities/wife.yaml → load only that file.
  • Without this: the loader has to either dump whole layers (expensive) or guess (lossy). With it: surgical lookups, predictable cost.

So no, that's a distinct feature and worth its own treatment. I'd propose adding a Sketch 5: The Roadmap — showing:

  • An index.yaml per scope, structured as a lookup table (id → path → summary → tokens)
  • A query example: "Character needs to recall topic-x" → scans the indexes → finds the path → loads just that file
  • The savings math (e.g. "47 files × 800 tokens = 37k tokens" vs "1 index × 600 tokens + 1 file × 800 tokens = 1.4k tokens")

I'd also tweak Sketch 3 to show the loader doing a two-pass read: pass 1 = indexes only (always loaded), pass 2 = specific entries (on demand).
