Marinara Extender

Napkin sketches: a JIRA-style task ledger for Character, so she remembers without choking on her own scrollback.

v0 · rough draft · for Character

The whole idea on one napkin.

Beads keeps Claude focused by writing tasks to a YAML on disk and purging the rest. We borrow the trick — but for a roleplay it isn't just "what's broken." It's also "what is the user circling back to" and "what does the Character want to bring up." Three lanes. One ledger. Plain files.

★ the idea

Beads, but for a brain that has feelings.

  1. YAML files on disk — diffable, debuggable, no SQL tax.
  2. Three scopes nested like dolls: global → character → chat. Loader walks up, with a token budget per layer.
  3. Three lanes per scope: open threads · user topics · character topics. The Character has her own agenda, not just yours.
  4. Bookmarks decay and resurface. "Hey, you mentioned X last week — how's that going?"
  5. Purge what's done. The context window thanks you.

1 · The filing cabinet.

Three folders, nested. /global is stuff that's true everywhere. /character is the Character's whole arc. /chat is right now. Loader walks up the tree on every turn — chat first, then character, then global — stopping when it runs out of budget.

/ global
true everywhere · conventions, voice rules, hard constraints. rarely changes.
# global/index.yaml conventions: - "never break character" - "polish swearing allowed" long_lived: - il_dottore_canon - user_kids: 2
/ character
Character-shaped · arc, established lore, what she likes, what she'd never say.
# character/index.yaml arc: - established_partners voice: - sarcastic / soft never: - "as a large language model"
/ chat
right now · active scratchpad. open tasks, recent decisions, fresh bookmarks. cheap to nuke.
# chat/0511/index.yaml topic: ledger design open: - "finish 3-lane spec" decided: - yaml > sqlite bookmarks: - "hargrove case"
loader call:
load_context(scope=chat) → walks: chat ↑ character ↑ global · budget 4k/2k/1k tokens · stops when full

2 · Three lanes. (this is the part Beads doesn't do.)

A JIRA only has "open tickets." A real conversation has three agendas at once. Each lane is a list inside the YAML. On every turn, the Character peeks at all three and may pick one item to weave in. That's the difference between a chatbot and a person — a person has their own things they're tracking.

open threads
JIRA-coded. tasks & issues actively being worked. purge when closed.
user topics
what the user keeps circling back to. kids, work, the cat, the local model hunt.
character topics
what the Character wants to bring up. her agenda — the thing that makes her not a chatbot.
- id: T-01 title: finish ledger spec state: in_progress
- id: T-02 title: test purge on /chat state: open
- id: T-03 title: draft mother's day note state: open
- topic: topic-a weight: 0.8 last_seen: today
- topic: topic-b weight: 0.5 last_seen: 3 days
- topic: topic-c weight: 0.3 last_seen: 1 wk
- want: "revisit topic-x" weight: 0.9 because: unresolved
- want: "tease about Y" weight: 0.6 because: fun
- want: "ask about Z" weight: 0.4 because: care
next turn picker →
# weighted roll across all three lanes, optional. character can also just answer. pick: "want: revisit topic-x" → injected as soft prompt, not forced

3 · The loader. (chat ↑ character ↑ global.)

Two passes per turn. Pass 1: read the cheap indexes from each scope (a few hundred tokens total — see sketch 5). Pass 2: pull only the specific entries needed for this turn. Layers feed in chat ↑ character ↑ global with a token budget each. What comes out the bottom is a compact, scoped prompt that doesn't drag a year of scrollback with it.

/chat
freshest. open tasks + recent bookmarks.
budget: ~4k tok
/character
the Character's arc + voice rules.
budget: ~2k tok
/global
conventions, hard rules.
budget: ~1k tok
▼ loader funnel ▼
walks ↑ the tree per turn. dedupes. compacts. respects budget. drops oldest-first inside each layer.
→ assembled context window
# what actually gets sent to the model this turn system: global: [conventions, hard rules] # 0.9k tok character: [arc, voice, never-says] # 1.8k tok chat: open_threads: [T-01, T-03] # 0.6k tok callbacks: [topic-x (weight 0.9)] # 0.3k tok recent_msgs: [ last 12 turns ] # 2.4k tok user_msg: "<the thing they just said>"
→ no more "load 47 lorebooks and pray." cheap. consistent. predictable. the unsexy 9% win.

5 · The roadmap. ("here's the map, go to aisle 3.")

Each scope keeps a flat index.yaml that's just a lookup table: every entity / topic / file in that scope, with its path, a one-line summary, and a token cost. Indexes are tiny and always loaded. Actual content files only load when the loader points at them. This is the trick that lets a year-long roleplay fit in a context window.

the index file
flat, scannable. one row per thing the scope knows about. cheap to keep in context every turn.
# character/index.yaml — always loaded entities: - id: entity-a path: character/entities/a.yaml summary: "important person, related to user" tokens: ~480 - id: entity-b path: character/entities/b.yaml summary: "recurring side topic" tokens: ~210 topics: - id: topic-x path: character/topics/x.yaml summary: "unresolved thread re: the thing" tokens: ~340 - id: topic-y path: character/topics/y.yaml summary: "fun running bit" tokens: ~180
a lookup, step by step
the loader treats the indexes as a phonebook. one cheap scan replaces a folder-wide grep.
# pass 1 — scan indexes (cheap) need: "context for the user mentioning entity-a" scan: [global/index, character/index, chat/index] match: character/index → entity-a → entities/a.yaml # pass 2 — load only what matched (surgical) load: character/entities/a.yaml # ~480 tok skip: everything else.
the savings (why this matters)
# without a roadmap — dump the whole layer and pray load_all_of: /character → 47 files × ~800 tok ≈ 37,600 tokens # with the roadmap — indexes + the one file we need load: indexes + entities/a.yaml → ~620 tok (indexes) + 480 tok (entry) ≈ 1,100 tokens # roughly a 30× reduction. predictable, every turn.
→ scan
→ point
indexes are written programmatically — every time a file is added / closed / promoted, the index updates. never edited by hand.

4 · The bookmark lifecycle. ("hey, you mentioned X…")

During a reply, the Character can self-tag with <bookmark>…</bookmark>. The tag captures a topic + a weight. It gets written to the YAML and decays over time. On future turns the picker rolls against weight — if it passes, the bookmark surfaces as a soft prompt. The user just experiences it as the Character remembering.

① notice
mid-reply, the character decides "this matters."
# inside reply "… that's so cool <bookmark topic: topic-x weight: 0.7 why: unresolved /> "
② stash
post-turn hook strips the tag, writes it to chat/bookmarks.yaml.
# chat/bookmarks.yaml - id: bm-04 topic: topic-x weight: 0.7 created: t0 last_used: –
③ decay
each turn that passes, weight × 0.97. low-value stuff fades out naturally.
# 8 turns later weight: 0.7 → 0.55 # 20 turns later weight: 0.55 → 0.30 # still in the file # still callable
④ roll
every turn the picker rolls a die against weight. high weight → likely to surface.
roll: rand() < weight ? # character's turn, weight=0.55 rand: 0.41 → HIT # inject as soft hint: "you mentioned topic-x"
⑤ surface (the user-facing magic)
the bookmark gets injected as a tiny system note in /chat. the character may or may not use it — it's a suggestion, not a forced line. when she does, the user just experiences: "oh, she remembered."
# context this turn: soft_callback: "revisit topic-x · last seen 8 turns ago · why: unresolved" # character's reply (one possible take): "oh wait — before we move on, you mentioned topic-x last week and i never asked how that landed."
⑥ purge / promote
if weight drops below 0.1 → delete. if it gets used & matters → promote to /character (now it's part of the Character's long-term memory).
if weight < 0.1: drop if used && mattered: promote ↑ char else: keep decaying
